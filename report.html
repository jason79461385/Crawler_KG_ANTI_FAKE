Final Project Report · 防詐情資控制台

Scam Intel Console

一個聚焦於台灣常見詐騙情境的防詐 demo,整合「多來源即時爬蟲 ▸ SQLite 持久化 ▸ Neo4j 知識圖譜 ▸ 規則 + Embedding + LLM 混合分析 ▸ SSRF 防護網址驗證 ▸ Markdown 對話助手」六大子系統,在單一可執行的 React + Express 全端原型內呈現完整資料流。

專案類型 · 全端 Web Prototype 技術棧 · React 19 · Vite 7 · Express 5 · TypeScript · Neo4j · SQLite 報告日期 · 2026-05-19 版本 · v0.1.0

一、專案動機

詐騙手法在台灣演化極快,從 LINE 假投資、假交友、解除分期、165 簡訊、釣魚連結,到結合社交工程的多階段攻擊。 一般使用者要在訊息瞬間做出「這是不是詐騙」判斷,缺乏資料支援與即時佐證。

本作品目標是把多元防詐情資(官方 165 案例、社群 PTT、Google News、PhishTank/OpenPhish 釣魚 feed)整合進一個可互動的控制台,讓使用者:

  • 看見即時案例知識圖譜(誰、用什麼平台、騙什麼)
  • 輸入可疑訊息,得到 混合式風險評分 + LLM 解釋
  • 驗證可疑連結是否在已知釣魚清單或啟發式高風險特徵內
  • 透過浮動聊天助手用自然語言互動,回覆完全以 Markdown 排版
  • 主動回報遭遇,讓系統長期累積本地差異化資料

四、技術亮點

① 全鏈路 Graceful Fallback

Neo4j / LLM / Embedding / 任一爬蟲故障,系統都不會炸 — safeRun + is*Configured() 把外部依賴包成可選層,單來源失敗只標記 live:false。

② 混合風險評分

規則式關鍵字 × 實體比對 × Embedding 餘弦相似度 (×42 權重) × LLM JSON 輸出,四層加總並 clamp 至 0-96。任一層失效系統仍可運作。

③ SSRF 多層防護

阻擋 loopback / private / link-local / CGNAT / ULA IPv6 / metadata.google.internal,並做 DNS rebinding 後解析驗證,加 timeout、maxBytes、maxRedirects 三道閘。

④ Neo4j 增量 MERGE

從原本「MATCH (n) DETACH DELETE n 全砍重建」改成 MERGE + lastSeenAt 標記 + 過期清理,並加 60 秒 retry backoff 避免單次失敗永久禁用。

⑤ 知識圖譜三層快取

伺服器端 in-memory cache + ETag/Cache-Control + 前端 SnapshotProvider prefetch,讓首次 /graph 從 6.7s → < 200ms。

⑥ 165 SPA 反向工程

165 全民防騙網是 Angular SPA,純 HTML 抓不到內容。逆向 JS bundle 找到真實 API CIB_DWS_API/api/CaseSummary/GetCaseSummaryList,成功抓到 160K+ 案例。

⑦ 內容過濾器

BLOCK / ALLOW 雙清單區分「政治新聞名字含詐騙」vs「真實案例」,避免 PSA、政黨議題、統計排行汙染 KG 與分析語料。

⑧ Docker Compose 一鍵起 Neo4j

npm run dev:full 會跑 db:up 自動偵測 Docker daemon、port 衝突、容器是否已起,再等 bolt + HTTP 7474 雙就緒。冷啟動約 8 秒,熱啟動 ~1.3 秒。

⑨ 浮動 Markdown Chat Widget

右下角圓形入口,localStorage 持久化對話歷史,react-markdown + remark-gfm 渲染標題 / 表格 / 條列 / 程式碼 / 連結。LLM 模式與本地規則模式自動切換。

⑩ 熱過濾的 KG 互動

類別 chip 切換採 vis-network DataSet.update + hidden flag,新增類別後節點立即出現/隱藏,不需刷新頁面;physics 短暫重啟 700ms 重新排版後自動關閉。

六、風險評分核心公式

score = min(96,
  10
  + matchedKeywords.length * 8
  + matchedEntities.length * 10
  + scoredEvidence.length * 14
)

# 單篇 evidence 分數
evidenceScore =
  overlapEntities * 24
  + keywordHits * 14
  + cosineSimilarity * 42   # 僅當 embedding API 已設定

level: score >= 70 → high
       score >= 40 → medium
       otherwise   → low
常數調諧備註: 24 / 14 / 42 / 8 / 10 / 14 / 70 / 40 / 96 全為人工 tune,改一個權重需連帶檢視其他,避免區間漂移。

十二、結語

本作品在 prototype 階段以「完整資料流」為核心目標 —— 從多元爬蟲、SQLite 持久化、Neo4j 增量同步、SSRF 防護、混合風險評分、浮動 Markdown 對話,到熱過濾知識圖譜, 每一層都做到「外部依賴可選 + 任一失敗 graceful 降級」的工程紀律。

要走向 production,下一階段最重要的是: (1) 把規則 / 關鍵字外部化讓非工程師也能維護; (2) 持續累積使用者回報,讓 embedding 檢索的語料規模從現在的 ~30 篇成長到 1000+ 級別; (3) 把 165 / SafeBrowsing / PhishTank 等官方來源轉成 production feed, 搭配 prompt caching 把 LLM 成本壓低,讓系統真的可以開放給一般使用者每日使用。
